Cloud Build and Deploy Script

 

So far things look like they are going well.  The important thing to note at the moment is that q2c-web-app is populated with the application and support files needed to get your sample application into the IBM Cloud.  The next step is to have Quick2Cloud generate a script filled with a sequence of IBMCloud and Quick2Cloud Commands that will get the application up into, and running, within the IBM Cloud.

 

The command sequences are:

 

1.       npm Install

2.       cf login

3.       cf push

4.       cf env

5.       q2loud

 

NPM fetches any required Node.js modules needed by the application then installs the modules in the application directory.  The script then logs-in to the IBM Cloud, pushes (copies) the entire directory, except node_modules, up to the Cloud where it is Built then Deployed to be a running application.  Next, the script gets the applications Environment Variables using the ENV command then the q2cloud command parses them and creates two files that the CFENV function reads; all designed to make these variables available to your locally running application.  Since this script will be stored in your q2c-web-app directory it can be used over and over as you make code changes followed by a wish to redeploy.

 

Command Shells

 

Tip: A word or two about Command Line Interface Shells.  You will most likely be using all of them so keep them as a shortcut on the desktop

 

With the installation of the IBMCloud Commands you did when fulfilling prerequisites, you should have several command line shells to choose from:

 

1.       DOS Command Prompt (cmd.exe)

2.       PowerShell (Powershell.exe)

3.       BASH (git-bash.exe)

 

Locate each executable and make it a shortcut on the Desktop.  Trust experience, this will make your life much simpler.

 

The only work you will need to do outside of the browser interface is to use one or more of these shells to execute the generated scripts that invoke individual commands to initiate the deployment into the Cloud.  You need to do this step manually because the browser, for security reasons, does not autonomously execute Shell scripts. 

 

You can pick anyone of the shells to run the Quick2Cloud scripts but in our course we will focus on the DOS Command Prompt due to its ease of use.  PowerShell can be used with some caveats.  BASH, for the purpose of our college, will be used when working mostly with repositories and SSH keys and that starts in Course 103.  But ultimately, it will be your choice of shell.

 

Caveat: PowerShell can be governed by corporate policy and that policy may prevent the running of scripts.  If this occurs, you will need to issue the PowerShell command get-help about_signing where it will tell you how to override that policy if your company allows it to be overridden.

 

Else, outside of using BASH for generating repositories and SSH keys, you can always use the DOS Command Prompt.  In the worst-case scenario, you can always cut and paste from the script and run each BASH Command one at a time.

Step 6A

 

You can start the application Build and Deploy over!

 

One of the key features of Quick2Cloud is everything you do can be started over.  If you did not get the sample web server to run in the IBM Cloud go back to the Main Menu and try it again.  This time look closer in the shell output to see if the Cloud issued any error messages. 

 

Most problems to look for in the shell output are:

 

Application did not start because you ran over your allotted memory quota.  This happens if you have other applications running and the sum total of them puts you over quota. 

 

 

Keep in mind that with the LITE no-cost IBM Plan being used by Quick2Cloud; it might put you over quota on memory if you exceed 512M (which would happen on your 3rd running application).  By default, each application is allotted 128m with the sum total of all running applications needing to be less than or equal to 512M.  Applications that are stopped do not count toward your memory quota.

 

Another issue to go wrong, you may have edited and inserted a bug in the sample application and it crashed causing the URL to not be active.  You will edit the application later but dont edit it as of yet.  If you did edit, then go back to the Main Menu and start Course 100 over regenerating everything but this time dont modify the code.  But first, clear out your q2c-web-app directory of all files and sub directories, even the hidden ones.  Best way to accomplish this is delete the q2c-web-app directory and recreate it. 

 

Note: q2c-web-app does have hidden directories.  That is why you should delete it to clear it.

 

If everything went well you should have a running web application in the IBM Cloud as shown by the URL rendering the sample applications home page.

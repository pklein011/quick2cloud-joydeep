Clustering Containers using Kubernetes

 

Microservices

 

In your previous college course, you ported your Quick2Cloud Sample Web Application to a Docker Container and saw the value Containers bring to the table.  However, the use of Docker Containers really has major importance when you begin the process of containerizing various parts of your application, not just the whole application, making the additional Containers you create really microservices that your application uses.

 

The ability to carve your application into various microservices provides a lot of advantages.  For one, well defined microservices can be reused by other applications in a kind of off the shelf way.  For example, assume you removed the Cookie processing from your sample application and made it a microservice with HTTP or REST interfaces; it would act like a Cookie processing web server that any application could call by providing its IP:Port address.   And, through the use of Docker Containers, you could build a repository of microservice images, place them all in a registry then make them quickly available to your application using a simple Docker Run command.

 

There are other advantages of architecting your application to use microservices, they run asynchronous to your application, can be load balanced such that if one fails the other takes over, providing your application extreme performance and reliability.  Being separate, these microservices can be developed independent of your application allowing you to divide responsibility to different group members who may be working with you on your project.  And, you can run the microservice containers across systems making your application and its services part of a rather large distributed Container Swarm.

 

 

As the use of Docker was adopted by IT and microservices started becoming the latest in a growing trend of Cloud architecture, the number of Docker Containers naturally grew as can be seen in the trivial example above (in that proposal you went from two to three Containers).  In prior courses, just your application and database were required but as you carve out pieces of the application and convert them to microservices the number of Containers can become large.  You can only image how many Docker Containers one could carve out if the application was of a large size.  The answer is anywhere from a few, to dozens, to hundreds and even into the thousands, especially if you create redundancy in your microservices using load balancers.

 

You can now imagine the management headache that comes into play trying to orchestrate a large swarm of Containers not to mention that many of them will have to run on other systems to distribute the load should this application become popular.  In reality, you, as an individual, may never exceed the number of containers that you, by yourself, can handle but should you be part of a group building an important and large piece of software; managing these containers will quickly become an irritating roadblock.  The good news is that there is an architecture available for orchestrating these container swarms.  Actually, there are a couple architectures but in this course the focus will be on the one that is the most adopted by the Clouds.  That orchestration software and methodology are part of a solution called Kubernetes.

 

Clusters

 

Kubernetes is software that is an abstraction layer above Containers that is typically known as a PaaS (Platform as a Service) although Kubernetes is really a subset.  You can think of a Kubernetes environment like a virtual operating system running on top of whatever operating system is supporting it.  As a virtual OS it has abstracted components that your Containers will interact with.  Kubernetes goal is to orchestrate all your Containers such that you have a simple methodology for keeping track of and providing some amount of control over.  The heart of this methodology in the IBM Cloud is the Kubernetes Cluster; which is really a collection of Containers that can be run and managed in a simplistic way.

 

Kubernetes is a very large and somewhat detailed subject that will be explored in one of Quick2Clouds 300 series courses.  In this course, you will be introduced to IBM Clouds implementation of Kubernetes by porting your sample applications Docker Container into a Kubernetes Cluster.  Once it is in the Cluster and you have an IP:Port address, then the actual management of that Cluster will be addressed in that other course.  For now, you will proceed to getting your Container to be part of an IBM Kubernetes experience.

 

An IBM Cloud Kubernetes Cluster

 

 

Remember, IBM Kubernetes implementation is a virtual abstraction of what we know as classic OS architecture so if it looks a bit strange its just that you are seeing it for the first time.  Once explained, you will have a much better appreciation of how it all works. 

 

PODS

These are Docker Containers that are packed together in a Deployment.  Each container is isolated but can communicate with each other via a local API and network addresses that is exposed by Kubernetes.  Only the Containers can communicate with each other, the outside world is isolated unless a request comes in through the Service which may be configured to only talk to your sample application.  It is through the Services provided IP:Port that you will be able to communicate with your application from a browser.

 

Worker Nodes 

This is the compute power allowed to be given to your Deployment.  Think of the Worker Nodes as a set of Workers that are virtual CPUs that get assigned by Kubernetes to supply your application and microservices with the ability to execute.  The more Workers you have in the Worker Nodes, the more parallel work can get done.  In this course, you will start out with 1 Worker Node which has the responsibility of executing your sample application and everything else.

 

Worker Pools

Governs the total amount of Kubernetes compute power and storage available to all Deployments in the Cluster.  However, what compute power and storage provided to your Deployment depends on the Worker Nodes configuration.

 

Service

For your sample application and other Pods, controls communication from outside to the Pods.  All outside network traffic is managed through this resource.

 

Secret

Contains sensitive information needed to make communications with IBM Services that have been provisioned on your behalf.  This includes URLS, userIDs and passwords.

 

Registry

This is an IBM registry where your application and microservices images are published too.  Kubernetes will load from this registry and instantiate PODs (Containers) of your application and microservices after the Cluster is initialized.

 

Through the use of a DOS Prompt or PowerShell, you will be able to configure each of the pieces above to best fit your needs.  Later, with IBMcloud and Kubectl Commands, you will be able to do many advanced management items including having distributed Clusters with distributed PODS all the way to moving Clusters across Cloud regions.  Again, that will be shown in the 300 series courses.

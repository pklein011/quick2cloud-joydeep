Automatically Scale Instances of your Sample Web Application

 

Wow!  You are now running your Quick2Cloud Sample Web Application in the IBM Cloud, most likely with a repository and Continuous Delivery in place, and looking pretty professional too in that you are using technologies that makes your application reliable and of high quality. 

 

Your hard work has made it easy to develop features as often as needed, test those features in the Cloud, and then have your production application updated without any outage to your Customers.  That is pretty powerful stuff!

 

However, now that you have enabled an application capable of serving many Customers, which, if it should it become well-received, will need to handle the situation where large Customer demands over-burdens its resources, degrades its performance and possibly cripples its capability.    And, with all the work you have put in, that would be a disappointment for such popularity you have created.  You need a way to beef your application up as its demand rises. 

 

One IBM Cloud instance of your application may not be large enough, resources wise; to handle this excitement you are creating, especially if your application gets high traffic rates, causing computing resources to grow out of control.  IBM Cloud provides a solution for this situation; a solution that allows you to scale your application instance to support the prominence of your new offering.  IBM calls this Auto-Scaling.

 

Note:  Auto-Scaling is the ability to automatically expand and contract the number of application instances to provide best performance for your Customers

 

You accomplish this by creating a set of Auto-Scaling Rules that instructs the IBM Cloud to increase or decrease the number of application instances you have running based on a set of monitored performance metrics reporting the health of your application.  If an Auto-Scaling rule is triggered, the IBM Cloud will automatically create additional application instances on your behalf.  When monitoring indicates that the extra instances of your application are no longer needed - the IBM Cloud automatically removes the excess instances bringing you down to a number best suited for the most effective performance.

 

Note: Auto-Scaling Rules typically come in pairs.  One for increasing the instances and one for lowering

 

Let us take a look at how this works by examining the diagram below:

 

 

Initially, you have one instance of your sample application (qvsample) running in the IBM Cloud.  This can run as just a process by itself or it can be part of the Toolchain that is interacting with your application.  In either case, if that single application exceeds 80mb of actually used memory then the Auto-Scaling feature would kick in and create one addition qvsample instance balancing the request load between them.  In fact, this process would continue expanding until you had enough instances of qvsample such that used memory would be low for each so that every Customer was getting desired performance.

 

Tip: There are several metrics you can use to determine performance for Auto-Scaling other than used memory.  They are shown later in this course

 

Once the load on your application has resided, this can be due a reduction in demand by the Customers, it would be advantageous to stop the additional application instances since they may not be needed to help with performance anymore.  Auto-Scaling will examine metrics surrounding the application; metrics that are specified in the Auto-Scaling Rules, and when a metric falls below its threshold - additional instances will be removed until a point where the rule deems performance as acceptable.  This is illustrated in the diagram:

 

 

In the illustration above, as your Customer’s leave for the day the load on your application decreases.  Customers, at their peak, represented 275 people but now only 45 Customers remain.  Since IBM Cloud is balancing the load between all your applications instances, your Auto-Scaling Rules may determine its time to shut down some of the excess instances because they are no longer needed.   In the rules set above, when the used memory falls below 30mb, it becomes time to shut those instances down.

 

Note: Auto-Scaling uses more IBM Cloud resources and therefore may cost you more as the instances increase.  For the IBM Lite no-cost plan you may be able to scale to two instances.  After that you will have to upgrade to take full advantages of the IBM Cloud Auto-Scaling facility.  For this course, this example demonstrates its function.

 

It should now be obvious with Auto-Scaling,  it is fairly simple to scale your application to any number of potential Customers by letting IBM Cloud and Auto-Scaling Rules determine the best number of application instances to have running at any given time.  You can start out small in your quest for application greatness and know that as its popularity grows that using Auto-Scaling Rules you can handle any Customer load put on it.

 

To be completely correct in that last statement, increasing the number of application instances based on resource demand may have an effect on any connected Services such as a database.  Those Services would need to be looked at and their size and capability should be planned for based on what you think the largest Customer load would be. 

 

Step 0

 

Let us create two Auto-Scaling Rules for your newly deployed Quick2Cloud Web Sample Application to see how this all works.

 

To accomplish IBM’s Cloud Auto-Scaling, you need to see if you have the CF-Community plug-in installed.  The CF-Community plugin contains the feature that allows you to use the Auto-Scaling facility.

 

Issue the following command:

 

cf list-plugin-repos

 

 

If you see the CF-Community in the list then you are good to go.  Skip to Step 2

 

Step 1

 

You execute this step only if you do-not see the CF-Community in your above plugin list.

 

Issue the following command

 

cf add-plugin-repo CF-Community https://plugins.cloudfoundry.org

 

Step 2

 

Now you need to install the Auto-Scaling feature from the CF-Community of plugins.

 

cf install-plugin -r CF-Community app-autoscaler-plugin

 

 

Step 3

 

Let us check the Auto-Scaler’s regional endpoint to make sure it matches the region where your application is running.  For example, you find your regional endpoint by issuing:

 

cf api

 

 

Step 4

 

Notice the API has an endpoint of ng.bluemix.net.  That is the endpoint for the US-South region.  Let us find where the Auto-Scaler says it is located.

 

cf asa

 

 

Step 5

 

In this case, your endpoint and the Auto-Scaler endpoint are in the same region.  This is the desired state.  However, if you find the Auto-Scaler is not in the same region then you will have to direct it there.  This is accomplished by issuing:

 

cf asa autoscaler.ng.bluemix.net

 

 

You are now ready to define your first Auto-Scaler Rule

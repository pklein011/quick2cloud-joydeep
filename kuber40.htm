Clustering Containers using Kubernetes

 

Kubernetes Toolchain

 

If you took the prior course using Docker commands to create a sample application image in the IBM Cloud Registry, Quick2Cloud will assist you in getting that image quickly running in a Kubernetes Cluster.  However, that will occur a bit later in the course.  For now, you will continue as if you did not complete the previous course on creating a Docker image by you letting the IBM Cloud do it on your behalf.

 

To accomplish this, you will create an IBM Cloud Kubernetes Toolchain; your sample application code will be the input pulled from your sample application’s Github.com Repository.  The Toolchain will automatically dockerize your code and create a Docker image, then store that image into your personal IBM Cloud Registry.  Next, the Toolchain’s Deploy Tool will retrieve it from the IBM Cloud Registry and place your application inside your Kubernetes Cluster as a Pod and start it running.   As a Pod, it will coexist with the several other Pods in the Cluster, but your Pod will be the major application with the others acting as support programs.

 

Basically, the IBM Cloud allows you to skip the Docker process on your local Desktop by doing the dockerizing for you in the Cloud.  Let’s proceed by getting an understanding of how IBM Cloud does the Docker transformation to Kubernetes.

 

Before we start, your sample application must have a Dockerfile and .dockerignore file in its repository as pointed out in the previous course.  Quick2Cloud will provide you with them if you did not create these files in your previous work with Docker.  Generate these files and place them in your q2c-web-app working directory and do not rename or add extensions.

 

Generate:  cluster-docker-file

Generate:  cluster-docker-ignore-file

 

Now that you have these two Docker files in your working directory, you need to commit them to your repository.  Use the following commands to accomplish this:

 

1.       git add .

2.       git commit -m “Added Docker Files”

3.       git push github master

 

Your GitHub sample application repository is now updated with the Docker Files.

Step 16

 

Great!  You now have a Kubernetes Toolchain defined and running.  The difference between this Toolchain and the one you used in previous courses is the addition of the Docker Containerization step.  Let’s look at the Build, Containerize and Deploy Tools as they exist in the Toolchain for Kubernetes.

 

When you are ready, click on the Delivery Pipeline icon in the Toolchain. 

 

Note: You can always find the Toolchain by displaying the IBM Cloud Dashboard. From the left side drop-down select DevOps.  The Toolchains will be displayed.

 

Delivery Pipeline Display

 

To begin with, the Kubernetes Toolchain uses your Quick2Cloud Sample Web Server Application as the input into its process.   The Build Stage will find your sample code at Github.com then make a Clone of it in the IBM Cloud.  Next, it passes this code over to the Containerize Stage where it will convert your code into a Docker Image, using your Dockerfile, and store it in your IBM Cloud Registry.  Last, the Deploy Stage will retrieve the image from the registry and add it as a Pod to the Cluster you have defined. 

 

Notice that you did not have to have much Docker knowledge around creating Containers and storing them in the IBM Cloud Registry; the Toolchain did all that for you.  However, you had to place a Dockerfile and a .dockerignore file among your source code so that the Containerized Stage knows how to build the image. 

 

After you accomplish the Deploy, look at the logs from the Deploy Stage (the link shown in yellow above), you will scroll down to the bottom and notice the IP:Port address Kubernetes has assigned your sample application.

 

 

 

Click on this address and you should see your sample application’s home page.

 

 

Step 17

 

Congratulation!  You now have your sample web application running inside a Kubernetes Cluster in the IBM Cloud.   Make sure you take the 300 series course on Kubernetes to see all the things you can do now that you are inside a cluster.

Implementing Continuous Delivery for your locally developed sample application

 

Local development with Continuous Delivery

 

Another method of developing applications but still use Continuous Delivery is to code locally at the desktop, bypassing the use of the editing tools provided on the Toolchain that you learned about in the previous section.  There is a bit more technology involved in doing development locally, but, for some, the value of developing and testing this way works in their favor.  Specifically, developing locally has the following advantages over developing in the Cloud:

 

1.    Restarting the application is faster

2.    Debugging is far easier

3.    Unit testing is easier

 

When running locally, your local git repository is where your commits are kept and your working directory is where you run the locally developed application from.  To enable the Continuous Deliver aspect of locally developed applications, you have to incur the extra step of pushing your repositorys most current snapshot to the Toolchains git repository which is located and secured in the IBM Cloud. 

 

Originally you pushed your local git repository to Github.com and IBMs Toolchain cloned and made a copy of it in its Toolchain repository before deploying and running it the first time for the entire world to see.  As mentioned, prior, the Toolchain only makes this clone once and after that is complete it is your responsibility to manually update the Toolchain Repository directly with your latest features.

 

Updating that Cloud Toolchain Repository is harder than it sounds because it is behind some pretty secure firewalls but there are a few methods to accessing it with Quick2Cloud showing you one of the more common ways.  This method makes use of a public / private key exchange with the Toolchain Repository.  Dont fret; in just a moment, Quick2Cloud will assist you in doing this. 

 

SSH Key generation

 

As you can imagine, IBM will not let just anyone push code into your personal Toolchain repository so you have to prove it is actually you before this step can be successful.  The way this is accomplished is by generating a public & private key pair then giving the public key to the IBM Gitlab Server.  Think of the Gitlab Server as the gateway server into the IBM Cloud for everything coming in over the network from a git command.

 

When issuing a git command on your desktop, to push your local repository to the Toolchain Repository, with a little bit of luck IBM will believe that push request is actually from you since it has your public key.  How this works gets accomplished via some complex key exchanges between your desktop and the IBM Cloud; a bit of complexity that thank goodness, you dont have to worry about.

 

Step 0

 

SSH keys are created using the BASH console and not the DOS or PowerShell Prompt.  Quick2Cloud will generate a script that will generate the SSH commands for this purpose.

 

Click to generate SSH public / private key pair script

create-ssh-key-pair

 

Copy the script to q2c-web-app and rename it with an extension to .ps1

 

For this you must use the BASH shell.  Type:

 

./create-ssh-key-pair.ps1

 

 

Follow the SSH prompts and make sure you put in a passphrase when prompted.  Leave the default save location intact (just press Enter) because that is where all the other pieces of software on your desktop will go, by default, to find your private key.

 

Note: After running the above script, the clipboard contains the public key so dont lose it.  You will be pasting it into a settings page in just a few minutes.

 

If for some reason you lose the public key by overlaying it in the clipboard, you can get it back in the clipboard by issuing the following Bash command

 

cat ~/.ssh/id_rsa.pub | clip   

 

Step 1

 

You will now want to login into the Gitlab Server in the IBM Cloud that represents your region.  Issue the following command to verify which reason your Toolchains are in

 

cf domains

 

 

Notice the line that says us-south.cf.appdomain.cloud, what you are really interested in is the first part (us-south).  Depending on what region your Toolchains are running in will determine what that first part will be. In your college example this is showing us-south

 

Therefore:  go to a web browser and type in the following (or replace us-south with your own region):

 

Step 2

 

https://us-south.git.cloud.ibm.com

 

and login with your ibmID

 

You are now in the Gitlab Server in your IBM Cloud region.  You will now configure your profile in it.  On the top right, click on the drop-down arrow in yellow and then click on settings

 

 

Locate, on the left, the icon that looks like a key (shown in yellow).  Click on this icon.

 

 

Step 3

 

Paste your public key (from your clipboard) in the yellow field below.  Then give it a Title and press the Add Key button.  In doing this you have just told the Gitlab Server for us-south that your desktop is allowed to push local desktop commits to your IBM Cloud Toolchain Repositories.

 

 

Step 4

 

Let us test out the connectivity from your desktop.   In your BASH console type the following:

 

ssh -T git@us-south.git.cloud.ibm.com

 

It will ask you for your passphrase then it should welcome you.  If that is what occurs then you are good to go to the next part of our course.  If it did not work, make sure you used a capital T in the -T option then try again.  You need to make sure this connection works before you continue.

Setting up for using Docker Containers

 

Docker Containers

 

After installing Docker from their website and after restarting your Windows 10 Pro system, you should have a Docker icon now appearing on the desktop as well as an icon on your task bar. 

 

This is shown below.

 

 on the desktop…..   on the task bar…..

 

The icons indicate you have Docker Engine installed.  The engine is comprised of the Docker CLI commands, the Docker Daemon and all other parts that make up a running Docker environment.  By default, after the Docker install, the Docker Daemon (a server component of the Docker Engine) will be running in the background and that is represented by the small Docker icon in the taskbar. 

 

It takes a minute or so to automatically start the Docker Daemon after Windows starts.  You should see a message pop-up indicating Docker has started.  If you don’t, open the Docker Desktop and start Docker manually.

 

With Docker, the Docker Daemon process does the heavy lifting during the creation of Containers and images.  You will use a set of Docker commands, via the DOS Command Prompt or PowerShell to communicate with the Docker Daemon, that is why it must be running when you begin using Docker for containerizing your application.

 

Step 5

 

Verify that Docker is up and running.  To do this you will ask Docker for a list of Containers that are running on your Windows 10 Pro system.  Each Container will be a lightweight virtual machine and since you are a first-time user, you should not see any Containers running.   You issue Docker commands to communicate with the Docker Daemon and with the ps parameter it will respond with the running Containers.

 

docker ps

 

 

You should see a list showing no Containers active as illustrated above.  The fact that you got this display means you have installed Docker correctly.

 

Step 6

 

On the www.docker.com website, once you login, there are several sample images that can be run directly in your Containers on your Desktop.  To find the samples just search on the word dockersamples at the Docker site

 

 

You will see a list of images that are completely built, configured and ready to go.  One such image is a web server called static-site that returns Hello-yourname when you visit its URL.  Let us create a Container, as a virtual machine, on your desktop, that runs this static-site web server image you found at the Docker website.  With Docker commands, this is very easy to do.

 

docker run -d dockersamples/static-site

 

 

Dockersamples is the name of the registry at the Docker site that contains sample images.  Static-site is the name of the image that will be used to instantiate an application as a Container on the local Desktop.  Issuing the Run command causes a few things to happen.  First, the Docker Daemon checks to see if that static-site image is already cached in the Desktop Docker repository and if not, automatically pulls it from the Docker Hub. 

 

Second, it creates a virtual Container with static-site running as the application.  When complete, the Run command returns a token string that identifies the Containers ID. 

 

Note: In the display above, the static-site image was already in my desktop Docker Repository.

 

Now, when you list out the Docker Containers on the Desktop you should see the static-site application running.  Notice the Container ID matches the ID created when the Run command was issued above.  Also notice this all took less than 10 seconds to happen.  That’s because you did not have to build and containerize the web server program – it was already made into an image by someone else a long time ago.  Now, wasn’t that fast way to start an application?

 

docker ps

 

 

Step 7

 

You now have Dockersample’s static-site web server running in a Container.  However, it shows above that the application uses two ports, 80 and 443 but by default does not show you the Docker Container Ports that map to them.  You will now stop and remove the Container from your desktop and re-run it with some common Docker parameters that will let you set and display some of the more important Container information.

 

docker stop 5ff7

 

docker rm 5ff7

 

You don’t have to use the entire Container ID in the Docker commands, rather only enough of the ID to make it unambiguous to any other Container IDs.   You will now start that Container again using the following Docker command

 

docker run --name static-site -e AUTHOR="Paul" -d -P dockersamples/static-site

 

 

The parameter --name allows you to name the Container any name you want.  Using the -e parameter says you will pass the AUTHOR as an environment variable containing Paul to the Container’s application and the -d says for the DOS Prompt to disconnect from the Container when it is done.  Should you omit the -d then the DOS Prompt will become static-site’s console log showing messages as it processes.  Most important is the -P parameter which asks the Container to list the Port mappings so you can see what Container Ports maps to the application’s 80 and 443 Ports.

 

Let’s now look to see the Port mappings of the Container

 

docker ps

 

 

Notice, to send a request to Port 80 in the Container’s application you would actually send that request to Port 32771 of the Container with the Container doing the mapping to Port 80 as it passes the request through.  The same type of mapping applies to Port 443 as well.  This Container mapping capability allows you to share the same Port numbers across many applications by mapping them to unique Container Port numbers.

 

Step 8

 

To verify that the Container application is working as advertised, you want to paste its IP:port address into a web browser and you should get the following Hello-Name result.

 

localhost:32771

 

 

Step 9

 

The real power of Containers is that you can instantiate more than one Container on your desktop or Cloud.  In fact, you can create a second Container using the same static-site application without need for application code changes.  Let’s call the second instance static-site-2 and let’s specify Port 8888 as a Container mapping for its Port 80 so it won’t bump into the same Port as the first one.

 

docker run –name static-site-2 -e AUTHOR=”Paul” -d -p 8888:80 dockersamples/static-site

 

docker ps

 

 

Wow, you now have two Containers running, both running the same application, but each is using a different Container Port for their application’s Port 80.  Again, if you pasted static-site-2’s IP:Port into a web browser you should see the Hello-Name response.

 

localhost:8888

 

 

Think of the possibilities of what Containerization can bring to your application.  But the real value comes when you take your Quick2Cloud Sample Web Server application and create an image then place that image in a repository where it can just as easily be run by not only your desktop but by a Cloud as well.  All that is required to run your sample application, anywhere, is a Docker Run command with your application image as the final parameter.

 

In the colleges next course, you will be assisted in building a Docker image of your Quick2Cloud sample application, run that image locally and get it into a registry for running in the Cloud.  But for now, you should have a good idea how the basics of this Container business works as you prepare to dockerize your sample application.

 

To understand everything that Docker is capable of, look at the Docker command reference and try each command out on your Hello-yourname Containers.

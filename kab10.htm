Appsody - a developer’s environment

 

Up until now, in your college courses on Containers, building these Containers were always followed by running them locally on the desktop or optionally using a Toolchain to place the image into Kubernetes Clusters in the IBM Cloud.  There is a cycle to this type of development, each time you make a modification to your sample application you must re-build your Container then re-run to verify the modification worked.   If the change was ineffective, another modification must be done and thus the cycle repeats.

 

Container Development Cycle:

 

1.       Make code modification

2.       NPM installs

3.       Issue Docker Build (new image)

4.       Build’s Housekeeping

5.       Issue Docker Stop (old Container)

6.       Issue Docker Run (new Container)

7.       Test and if bad go to step 1

8.       Else, continue next capability at step 1

 

You follow this cycle until all the code changes are compete, correct and your application features are ready for production.

 

Even though Docker is very efficient in its Build process, for a good size application the above steps can be a bit cumbersome.  At minimum, repeating these steps over and over for every code modification is tedious to say the least with a good chance of making a mistake while mindlessly following the order of things.   Partly born out of this tedium, partly for other reasons too, some technologies were developed that alleviates much of this repetition.  Enter Appsody, technology for Container development and application testing.

 

Appsody methodology begins by separating the source code from the runtime Image and treats them independently.   An Appsody Image has the ability, on its own, to run as a Docker Container and fetch your sample application code at its Container runtime.  Think of this as a Container running a special program whose job it is to dynamically grab your source code when the Container starts.  Not only does it fetch your sample code, but should you make a code change to your application, the Appsody Container gets notified of the change and fetches and reruns your application all over again.  This means, you build your Appsody Image one time, start it in a Container, and from that point forward all you have to do is modify your application code in its desktop directory and test the results – bypassing the repetitive cycle shown above in purple.

 

Here is what Appsody does for you:

 

 

In the Appsody development cycle, you start by initializing an Appsody development environment by fetching an Image from the Appsody Image Repository (1).  In your case that will be a Node.js Image.  Appsody’s running Container will automatically fetch your sample application source code (2) and place it into the Container and begin its execution.  As a tester, you will use a web browser and interact with the sample application looking for errors (3).  Should you find one, you will make a code change again (4) causing Appsody to automatically be alerted and re-fetch your sample application code and start it running again. 

 

When all is deemed working perfectly, you will issue an Appsody Build Command (5) that will package up the Appsody Image and your application code into a finalized Image that gets stored in Docker’s Local Repository and ultimately pushed into your IBM Cloud Registry (6).  Once in the IBM Registry, it is available to be placed inside a Kubernetes Cluster in the IBM Cloud.

 

Therefore, Appsody provides the developer a slightly automated containerized development process which is much quicker to develop and test in, without the repetition and time it takes to accomplish the task with Containers alone.  Also, because Appsody maintains a separate image from your code; that opens a whole area of development that allows for the supply of well tested public Images that can be reused by the developer in building their application.

 

Kabanero

There are other software contributors supplying public Images besides Appsody.  Kabanero is an Open Source entity that provides high performance images based off IBM’s Red Hat Linux.  Since you are getting an image from a public source, there may be concern about the quality and virus free nature of that image.  These Kabanero images are guaranteed to be of very highest quality and constantly checked for viruses.  In our course we will be using the Kabanero image for Node.js that is based on Red Hat Linux to run your sample application.

 

Let’s understand how using public images is an advantage.

 

Suppose a video streaming company produced a Kabanero Image that contained some of its proprietary cinema capability with Red Hat Linux and placed it into Kabanero’s public registry.  You, as a developer, could pull that Image and instantly have movie functionality for your application.  To develop, just run the Image as an Appsody Kabanero Container, have Appsody automatically fetch your application source code and have that code call the Image’s video functions.  Immediately, you begin testing, and as you fix errors, Appsody automatically re-fetches your application code, restarts, then your testing resumes instantly.  Within a short period of time you would have a highly functional application with video streaming services and with very little effort on your part.

 

You may be thinking, why don’t I create a video service without Appsody or Containers by just adding the video modules to my local desktop directory so the Cloud could build an executable as it did for me in Quick2Cloud course 100.  You could do that, however, and this is an important point, you may not have the video modules, may not know where to get the modules, not understand what modules you would need and the video service who owns the intellectual property may not want to give the modules directly to you.  Embedding the video service in a Kabanero with Red Hat Linux Image is a way for the video company to feel safe in sharing their technology.   And, face it, do you really want to have the burden on how to build the video modules yourself?

 

The best way to really understand what Appsody and Kabanero are all about is to use them.  In understanding them, you will place your sample application into an Appsody development environment using a Kabanero Image Stack, then place them into a Docker Container, and ultimately into your Kubernetes Cluster that you created in the prior course.

 

Step 0

 

Download and install the APPSODY commands.

 

https://appsody.dev/docs/getting-started/installation/

 

Follow the install for Windows

 

Step 1

 

Create a new directory on the Desktop Root called q2c-web-appsody.  Create a project directory called sample2cloud

 

md q2c-web-appsody

cd q2c-web-appsody

md sample2cloud

cd sample2cloud

 

You should now be in the q2c-web-appsody/sampl2cloud directory

 

List out the Images that are publicly available for you to use.  Don’t worry if you don’t see an image that has everything you need in it.  You can start with a base image and build on top of it until it is exactly what is required.  With Appsody, you can save away your customer built image into a local repository and have your peers use it as a starting point for their projects.

 

Step 1a

 

appsody list

 

 

The image you will be interested in for your sample application is the one in yellow called kabanero/nodejs.  

 

Note:  If you don’t see any Kabanero Images in your list then you will need to add Kabanero to your Appsody repository list.  You only need to do this if you don’t see them in your current list, otherwise skip this.

 

appsody repo add kabanero https://github.com/kabanero-io/collections/releases/latest/download/kabanero-index.yaml

 

You have now added kabanero to Appsody.  Re-run Step 1a and continue.  You should now see kabanero in the list.

 

Step 2

 

Initialize your sample2cloud directory with this kabanero node.js base image.  If the image is not already in your local Docker Repository, then it will be pulled from the Appsody repository and cloned in your local Docker Repository.

 

Note:  Use lowercase “k” in kabanero when referencing it or Appsody won’t find it.

 

appsody init kabanero/nodejs

 

 

Step 3

 

List out the sample2cloud directory to see the project files that were given to you.

 

dir

 

 

Notice the files in yellow.  They belong to the Hello sample application given to you by the appsody init command.  You will be modifying this directory by adding your sample application code.  The package.json file will be edited to name your qvsample.js as the application to start when the Container is created.

 

Step 4

 

Copy your q2c-web-app directory contents, and all sub-directories, into the above q2c-web-appsody/sample2cloud directory.   Use Windows File Explorer to do the copy since it will copy everything from the q2c-web-app directory into q2c-web-appsody/sample2cloud. 

 

Important: SKIP ANY FILES THAT SHOW A CONFLICT by FILE EXPLORER during the COPY

 

List out the directory again and make sure the copy completed successfully.  Look for the qvsample.js file for verification.

 

 

Step 5

 

Using your desktop editor, edit the package.json file and replace app.js with qvsample.js.  Then click on Save.

 

 

Next, you will build out this project with modules for Express, CFENV and Cloudant database.

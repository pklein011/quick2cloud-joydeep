Fantastic! You have a sparsely populated Kubernetes Cluster up and running in the IBM Cloud. Its sparse because the Cluster has some infrastructure Pods running that help in its management. For example, your Cluster will have:

 

1.       Pod for Watching for troubled applications

2.       Pod for collecting performance metrics

3.       Pod for VPN support

4.       Pod for autoscaling

 

This is a diagram of the IBM Kubernetes Cluster that is being provisioned

 

 

In the diagram, this is just a subset of the infrastructure Pods provided in this provision in support of your sample application. Yet, what is missing is the Pod that contains your sample application and that is what you will provide in the next set of steps. But, before you can do that, you must create an IBM Cloud Repository that will hold the sample applications Docker image that will eventually be retrieved by Kubernetes and put in the Cluster along side the infrastructure Pods.

 

Step 7

 

Lets create an IBM Cloud Registry that will hold your sample applications Docker Image.

 

Login into the IBMCloud

 

ibmcloud login -a https://cloud.ibm.com -u ibmid@email.com

 

 

Enter in your ibmID password.

 

Step 8

 

You must ensure your region is set correctly.

 

ibmcloud cr region-set us-south (or whatever region you are in)
 

 

Notice that IBMCloud, upon return from the command, assigned you a registry name of us.icr.io. Remember this; for it will be important for later processing. Depending on the region you are in may determine what registry name gets returned.

 

Wonderful! you now have a place inside the IBM Cloud to store your Docker images.

 

An IBM Cloud registry entry has three parts to its name. The first is the registry name IBM returned to you. The second part is your IBM Cloud namespace and the third part is your application name. A Namespace is a unique area of the IBM Cloud where ibm stores things like your Docker images.

 

For example, an entry in the IBM Cloud Registry could be named:

 

1.       us.icr.io/namespace/quick2sample

 

When you push your Docker image to the IBM Cloud Registry it will be named in this format.

 

Step 9

 

It is time to create your specific namespace in the IBMCloud. This space does have to be unique so Quick2Cloud recommends appending your ibmID to the names beginning. For example, your namespace could be ibmid-myimages

 

Note: namespaces must be 4 30 chars, lowercase, numbers, hyphens or underscores. No capitalizations!

 

ibmcloud cr namespace-add ibmid-myimages

 

 

You now have your first IBM Cloud namespace.

 

Be aware, this namespace has a finite amount of space associated with it, but you can upgrade out of the Free Plan for just a little extra cost. You should be able to store about 5 images before you max out due to the Free Plan quota.

 

Step 10

 

Lets take a moment and review your accomplishments. You have discovered that an IBM Cloud Kubernetes Cluster is a virtualized environment that contains components that help in the running and management of your sample application. Pods, for practical purposes, are nothing more than containerized applications that look a lot like processes in a classical OS like Windows. Pods are packed together in a Deployment, each Pod running as if it were oblivious to the Cluster environment. Along with your sample application are Pods that help maintain your applications health (the Watcher, the Scaler) and some Pods collect metrics to determine just how healthy you are (the Metrics).

 

Each Pod has a IP:port address and the Pods, being packed together, have API access between them. In a Cluster, the outside world is shielded and can only see in through the Service at its IP:Port interface. Pods execution are controlled by the Worker Nodes and together with the Worker Pools simulate virtual CPUs and memory as if the Pods were running in a classical OS.

 

Wow! Pat yourself on the back. You are playing in the major leagues now. When you find out later what amazing things you can do with these Clusters you will feel all this work and understanding was well worth it. Good Job!

 

Step 11 (ignore if you skipped course on Docker)

 

In this step you will take your sample application Docker image created in the previous course and get it put in as an IBM Cloud Kubernetes Pod. Dont worry, if you skipped the course on Docker you can still use Kubernetes just fine; just skip Step 11 and 11A. However, if you did take the course on Docker, proceed forward as we put your Docker Container into the IBM Cloud Registry.

 

Lets build your Quick2Cloud Sample Web Application once again as a Docker Container. Begin my making sure you are in the q2c-web-app directory where you place your Dockerfile when you finished the course on Docker Containers.

 

docker build -t ibmID/node-sample-web-app .

 

 

You now have an image created in your local Docker Repository. Lets take a look at the image.

 

docker images

 

 

Your next step is to prepare the image to be pushed into the IBM Cloud Repository. Remembering that the IBM Cloud Repository requires a certain format for its entries, you will create a Docker Tag (alias) for the entry in your Docker Repository in the correct format for IBMs Registry.

 

Revisiting the format of an IBMCloud Registry:

 

1.       IBM registry/namespace/application

 

Remember, you created a namespace prior that looked something like ibmID-myimages.

 

docker tag ibmID/node-web-sample-app us.icr.io/namespace/sample2cloud

 

 

If you look at your Docker Repository now you will see the following:

 

docker images

 

 

The first entry was created when the Docker build command was issued. The second entry, an alias of the first, has its name constructed so it can reside in the IBM Cloud Repository. You will now push the image from the local Docker Registry into the IBM Cloud Registry.

 

Remember to replace the namespace with your created namespace.

 

docker push us.icr.io/namespace/sample2cloud

 

This is one of the names you must remember. Jot it down on your cheat sheet for recalling later.

Example, jot down whatever you substituted for (us.icr.io/namespace/sample2cloud).

 

 

Well Done! You have an image of your Quick2Cloud Sample Web Application in your IBM Cloud Registry. By residing in the registry, Kubernetes can load it and make it a Pod in your freshly created Cluster. Lets take a look at it in the IBM Cloud Registry:

 

ibmcloud cr images

 

 

Focus on the second registry entry, that is the one just built for the sample application. For the purpose of this course, ignore the first entry.

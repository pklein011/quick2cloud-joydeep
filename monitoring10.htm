Advanced Availability and Response Time Monitoring

 

In building your application, as it advances in complexity and function, there will be a tipping point to where basic availability and response time monitoring needs to be augmented with additional performance information that specializes in application detail and topology.  Should you deploy two or more IBM Cloud servers or combined servers and micro-services, you may want to start looking at performance by workloads several servers provide.

 

At this point we are stepping into the realm of Application Performance Management (APM); a rather large topic in the IT world and a topic that is most beneficial to having for understanding a healthy running group of related Servers and micro-services.

 

In using APM, you will want to name workloads and relate them to the Cloud Servers they use.  It has proven best practices to render this information in a set of eye pleasing graphical displays. 

 

Step 0

 

Here is a dashboard container view of all workloads running for an organization in the IBM Cloud.

 

 

Each container above is named for a workload and each of the icons inside represent the specific servers, components, messages and events that make up that workload using color to indicate their current state.  Clicking on any individual icon will bring additional information to why that icon is the color it is.

 

Step 1

 

For example, here is a Transaction Icon rendered showing response time for a Cloud transaction.  You can see by the colors that the response time over the day has been good but a few times it has gotten close to unacceptable.  To help understand why it would be of interest to understand the topology the transaction takes.

 

 

With APM, the topology is monitored as well showing what physical hops a transaction takes in your Servers and gives a bit of insight into why sometimes it is taking up to 5 seconds to get a response.

 

Step 2

 

In looking at transaction topology, it can be seen that the HTTP Cloud server on the far left is the point-of-entry for this transaction.  The request works its way eastward, toward the right ended Cloud Servers, and along the way several of the in-between Cloud servers are indicating yellow which equates to marginal response time.  If you were going to help speed this transaction up, this topology indicates that the Trade S2, Trade S1 and Trade S3 Servers needs to be looked at; looking at other Servers would be more of a waste for speeding them up provides little relief.

 

 

Topology and response time are very good metrics but does not always tell the full story.  The APM story is also about how individual Servers are performing from an internal perspective.  APM looks at these internals as well by monitoring all of your Cloud Servers and micro-services.  In addition, you may have some Servers running locally in-house that participates in the Cloud server topology.  APM will monitor these Servers too, giving you a hybrid Cloud view of performance.

 

Step 3

 

Shown below is an example of those internal views across several of the effected Servers above.  You can see many different aspects of the Server performance that are measured.

 

 

This is just a small set of what is available with IBM’s APM solution.  Once you get your application going where it is depended on for your success it would be reasonable to try out APM, give it a trial run, see if it provides you with what is needed to reinforce your industrial strength offering.

 

For some, IBM’s basic Cloud monitoring will suffice.   For others APM fills their desires; what it really comes down to is what is your monitoring needs.

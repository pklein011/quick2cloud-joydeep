Updating Production to your latest version

 

You have finally come to the very last step in Continuous Delivery and that is updating Production with all the fantastic changes you made in the latest development of your application.  After previously pushing changes, over and over, to the Development Toolchain, diligently testing them in the Cloud after each update, finally blessing these features for accuracy and quality - it is now time to update Production for the whole world to enjoy.

 

In Quick2Cloud, updating Production with Development code has been purposely made a manual operation as part of the safety criteria protecting you from accidentally (or maliciously) corrupting your Production version.  Maybe a better explanation should be given to address this philosophy. 

 

Instead of using two repositories, you could configure the Toolchains to share one repository, then use the master branch (for Production) and a new dev branch (for Development) making the process much simpler. 

 

In that methodology, all it would take is a simple push of a button in any of the git tools in the Development Toolchain and Production Toolchain would be updated instantly.  Although that sounds good on paper, it does open the possibilities for accidents since you would be one finger push away from corrupting your Production code.  In the real world, that is to close for comfort.

 

By separating the repositories, that corruption can’t happen and that is a good thing.  The process to update Production, as shown below, is purposely manual in nature but very safe in reality.  Hence, it is purely from the trials and tribulations experiencing many mistakes that Quick2Cloud advocates this manual method of updating your Production Repository.

 

Step 7

 

Begin the update

 

You will begin by pushing the Development Toolchain repository into the Production Toolchain repository so that Production, by way of a third party platform, Production will get the latest updates but retains its previous Production snapshot in case something goes wrong should it have to revert back.  Again, this is purposely a manual step making it almost impossible to accidentally update Production; and for further protection, every step in the process is protected by a passphrase that only you know.

 

Note: The process for updating the Production repository from the Development repository is the same weather you develop in the Cloud or locally at the desktop.  

 

That means, you will need to generate a SSH Key and update IBM’s Gitlab Server with the public key as was accomplished in the previous steps.  If you have not done that yet, go back to SSH KEY Generation and generate yourself a key.  Then return and continue with Step 7.

 

Your desktop will act as the 3rd party platform for this process.  Create a directory on your local desktop with the name

 

q2c-web-app-merge

 

Make sure it is empty.  If it has been used for this purpose before it won’t be empty even if it looks empty.  That is because of the hidden git subdirectories.  It is best to delete the directory and recreate it.  Also, for ease of operations, make sure you are in the desktop’s root directory when you issue the following command.  Like always, update your region and ibmID with the ones that reflect your environment.

 

This is how it would look for pklein011...

 

git clone git@us-south.git.cloud.ibm.com:pklein011/simple-toolchain-pklein011-development.git  q2c-web-app-merge

 

 

Step 8

 

That command just populated the q2c-web-app-merge directory with an exact copy of the development Toolchain’s repository and placed it on your desktop.  This directory will automatically become a git repository after the clone is completed and will respond to git commands.  Now, you will define the Production Toolchain’s repository to the local repository by the following command in preparation for pushing up the new features to Production.

 

Change directory to q2c-web-app-merge

 

Here is the command to issue as if I was doing if myself:

 

git remote add qvprod git@us-south.git.cloud.ibm.com:pklein011/simple-toolchain-pklein011-production.git

git remote -v

 

 

Notice that qvprod points to the Production Toolchain repository and origin points to the Development Toolchain repository.   As you can see, you are defining to your local repository the from and too repositories.   It is very important that you spell the name of the Production Toolchain repository perfect or the update to Production will not work.

 

Step 9

 

It is now time to change your Production Repository to reflect your hard work in all that you have done in Development.  This is accomplished by pushing up the features into the Production Toolchain repository where the Pipeline Delivery tool will automatically restart your Production application making it available to the entire world to see.

 

git push qvprod master

 

 

Notice, beside the manual process, as a protection, at each step you are also required to enter the passphrase that you created when you generated your SSH Key.

 

Congratulations!   You now have Continuous Delivery configured along with a methodology to update your Production Toolchain repository in a simple and safe way.

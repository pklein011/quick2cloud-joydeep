Setting up for using Docker Containers

 

Why Containers?

 

We are going to kick your Quick2Cloud Sample Web Application up a notch by introducing it to the explosive new concepts of containerization and light weight virtualization.  You may ask why?  It seems to work just fine as a classic Cloud Application. Why then should I consider Containers?

 

There are many reasons containerization is popular and quickly being adopted by IT and Cloud vendors; some even say it is the future of computing.  Containers bring a new dimension to how you develop and package your application; dimensions you will come to understand shortly.  Thus, it is Quick2Cloud’s recommendation that you understand Containers, even if you don’t plan to adopt them yet, it could be important to your future as an IT professional.

 

Containers and its partner, Clusters, which you will learn about later, provide the following sets of values to your application:

 

 

Containers, Images and Registries

 

In your previous Quick2Cloud college courses, when you first pushed; then later created a Toolchain for your application, the IBM Cloud had to build an environment for you, on-the-fly, before it could deploy your application into the Cloud.  In the case of the Quick2Cloud Sample Web Server, the IBM Cloud had to provision a Node.js environment then download and install node modules CFENV and Cloudant plus a lot more. 

 

If you were to look at the CLI output in the DOS Command Prompt after your push occurred, you would see messages showing all the work the IBM Cloud was doing to build an environment for you.  Similarly, when you used Toolchains, the Build Tool did the building and if you looked at the Build Log you would see what it had to do to get your application ready for deployment.

 

There are a couple of issues with this classic Cloud approach.  Making the Cloud figure out and construct what environment you need is time consuming and with complex applications, those with lots of parts to them, could take the Cloud a long time to construct.  This makes turnaround of your application somewhat slow. 

 

Another issue with the Cloud doing the building, the Cloud may not always pick the exact levels of software you have been testing with on the Desktop, possibly giving you stability issues, especially if you run your application across Cloud environments.  As an example, your application may run fine on one Cloud vednor but not on another because the second vendor builds could have selected slightly different software levels.  This is especially true over time when new software becomes available and supersedes the ones you have been testing with locally.

 

With the introduction of Containers these problems disappear completely.  Containers allow you to build an image of your application locally, on the Desktop, then publish that image into an image registry.   As entries of the registry, images contain everything necessary for your applications to run.  You, the developer, pick the software levels and build configurations and everything else it needs during your image creation. 

 

Containers are lightweight virtual machines that load these images and then run them, bypassing the Cloud’s build stage, getting the application up quickly and without concern over Cloud build decisions.  And, since your Containers are completely self-contained virtual environments, they can run your image on any machine or Cloud environments and will use EXACTLY THE SAME software levels that you provided in the image build!  Because of all this, container concepts provide a lot of advantages when you start leveraging the Cloud or multiple Clouds with your application.

 

Another advantage reveals itself as a by-product of having to create images for Containers; many images are published publicly by developers and vendors, meaning, you have a public registry of well tested images that others have built and shared; perfect for you to take advantage of.  

 

As an example, you could load and run a Container with the image of a well-tested Mongo DB database in just a matter of seconds.  The Mongo image has already been built, configured and made ready to go.  You just locate it, yank it off the shelve, issue a container command and the Container’s virtual machine starts running.  Public images are like building blocks, they are services that can be quickly instantiated for your application to use without you having to build it yourself.  Think of the time savings you can obtain by the sheer volume of image reuse!  

 

You can find many public ready-to-go images after signing into www.docker.com.  Just select the explore link to start browsing.

 

 

Clustering

 

As you can see, an Image behaves as a template containing all the parts that make up your application.  A Container loads the image and becomes the virtual instance of that running application.  Should your application Container require communicating with another server, like a Mono DB database, it can load Mongo’s public image into an adjacent Container and pair the two Containers together as a Cluster.  That Cluster acts like a virtual machine containing your application and database; housing and orchestrating the environment for all the pieces that make your application work. 

 

Clusters can also be distributed and run on other machines or Clouds and when they are configured that way, they look like one virtual machine that spans many different machines.  One of the most useful features of a Cluster is that all members of the Cluster can be managed from a single management tool.  You will study about this orchestration tool in another college course.

 

It should be apparent that Clusters get a bit complicated quickly but don’t worry, Quick2Cloud will take these topics one step at a time in your college courses. 

 

Let’s start with Docker Containers

 

Note: IBM Cloud does not require a detailed understanding of Docker Containers for you to use their Clustering facility; a facility that is covered in another course.  IBM Cloud can automatically take your application and Dockerize it then places the Docker Container into a Cluster.  Quick2Cloud highly recommends continuing your work with Docker Containers so when Clustering is discussed you have a better understanding of what is going on.  But, should you decide, you can jump to the Course on Clusters and skip this course.

 

Step 0

 

For Windows you will need to be running Windows 10 Pro or higher.  Docker Containers requires the virtualization capabilities that this level of Windows provides.  You can also do this on Linux but the description here is based on using Windows 10; if careful, it can be adapted for Linux as well. 

 

Note: If you don’t have Windows 10 Pro or higher, or you don’t have a desire to learn Docker, you can skip this course and immediately go to the college course on Clusters.  However, Quick2Cloud highly recommends at least reading this section to get a better understanding of what is to come.

 

As part of the install, Docker will make use of your Desktop’s virtualization features by configuring these features as it requires.  You may see messages about Hyper-V and how Docker will make changes in this area and for most systems this will not pose a problem.  However, if you are already using applications that make use of Hyper-V, you may want to read up on how Docker may interfere with them.

 

Download the IBMCloud CLI Commands that operate on Docker and Clusters.  This is accomplished by issuing the following command in your PowerShell interface.  Open PowerShell in Administrator mode.

 

[Net.ServicePointManager]::SecurityProtocol = "Tls12"; iex(New-Object Net.WebClient).DownloadString('https://ibm.biz/idt-win-installer')

 

 

Run the installer and it may ask you to restart Windows.

 

Step 1

 

ibmcloud login -u ibmID@email.com

 

Setup the IBMCloud CLI commands for working with image registries.  This is accomplished by issuing the following command in the PowerShell interface.

 

ibmcloud plugin install container-registry -r 'IBM Cloud'

 

 

Step 2

 

Download Docker on your system by going to the www.Docker.com website (also called the Docker Hub).  Docker will download several pieces of software and may require a Windows restart.  After restart, issue the install again to make sure you did not miss anything. 

 

Step 3 

 

Log into www.docker.com and create a dockerID.  By getting a dockerID, by default, you will have a Docker registry granted to publicly store your images that you will be building.  Once your images are stored at Docker you can easily put them into Containers and run them in the IBM Cloud as well as the other Clouds.

 

Note: You can make your dockerID the same as your ibmID, although this is not required it may make it easier when working with Docker in the IBM Cloud.

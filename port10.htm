Porting Sample Application to Docker Containers

 

Dockerfile

 

Porting your Quick2Cloud Sample Application to run virtually in a Docker Container is simple now that you understand the basics of containerization.  The nice part of this work is that no code changes to the sample application are to be made to accomplish this incredible feat; only the introduction of some Docker configuration files in your q2c-web-app directory.

 

The heart of the conversion of an application to run in the Docker virtual environment is the Dockerfile that contains instructions to take your application and convert it into an image;  an image that at first resides on your Desktop in a local Docker repository then eventually in a public registry at the docker site.  You will also be able to use other registries that are more private to where you want to run; all that will be covered shortly.

 

With the build process, Docker uses this Dockerfile as input to the build process and when the build is complete it will be ready for use in a Container.  The commands in the Dockerfile that accomplish this are shown below:

 

Note: The Dockerfile must start with an uppercase D.  The IBM Cloud will ignore this file unless the D is capitalized.

 

FROM environment:<level>

 

Here you specify the runtime environment for your application.  In your sample applications case that will be the Node.js environment.  The level represents which release of Node.js you want your application to use.  You can replace <level> with a number of a Node release or you can use the word latest and Docker will build with Nodes most recent release.  Whatever software level you provide as the runtime, it will never change each time your Container runs.  It might only change when you build again and that depends if you change the <level> or you use the word latest.

 

WORKDIR /usr/src/app/<app>

 

Here you specify the directory where the build to do its work in.  You should replace <app> with your application name.  In your sample application build that name would be qvsample.

 

RUN npm install

 

For applications that need external modules to operate, they are listed here.   In the case of your Quick2Cloud sample application those external modules are CFENV, Cloudant and Express.  When specified here, Docker will build your image by pulling in those modules from npm so they will be available for your use when the Container starts your application.

 

Note: npm is a third-party library service that delivers modules to your application over the network.

 

COPY .  .

 

This tells Docker to copy all the files in your directory into the build.  If you have a .dockerignore file in the directory this will list the names of files you dont want to be copied.  Or, you could have a copy command for every file that is required for the build but that would be a tedious task.  Much better to let the wildcard copy shown above do the heavy lifting.  Dont forget the two periods at the end separated by a space.

 

Create a .dockerignore file in q2c-web-app with your Desktop editor and add the following statements.  Or, let Quick2Cloud generate you one for your q2c-web-app directory.  Save this file without renaming or adding extensions.

 

Generate:  cluster-docker-ignore-file

 

 

None of these .dockerignore files from your directory will be selected for the build.

 

EXPOSE 8080

 

Here Docker allows you to define the Port your application listens on in the Container.  You will need to map this Port to a Container Port when the Container is running so you can communicate with your application.  This is done on the Docker Run command.

 

CMD ["node","server.js"]

 

Finally, you specify the command used to start your application.  Both node and server.js have double-quotes around them. For your sample application your server name will be qvsample.js

 

Thats it!  You have just seen a basic Dockerfile with enough inside to instantiate your application in a Docker Container. 

 

Step 0

 

For your sample application, Quick2Cloud will generate a Dockerfile for you.  The contents of the Dockerfile are custom to your Quick2Cloud Sample Application but you can see from the Dockerfile commands above, it does not take much imagination to taylor these commands to any application you have. 

 

Generate:  cluster-docker-file  and place the Dockerfile file into your q2c-web-app directory without any modification to its name or extension.

 

Step 1

 

List out your q2c-web-app directory and make sure your Dockerfile has found its home in it.  You should see your .dockerignore file as well.  Fantastic!  You are all set to go.

 

 

Thats all it takes, two Docker files in your q2c-web-app directory and with a few Docker commands you will have your sample application containerized and running.

Dockerize your Sample Web Application

 

Step 2

 

Bring up a DOS Prompt or PowerShell and set it to be in your q2c-web-app directory.  Use your dockerID in place of whats shown and name your application node-web-app.

 

docker build -t pklein011/node-web-app .  (dont forget the period at the end)

 

 

If you look close to the messages generated by the Docker build you can see that CFENV, Cloudant and Express were all installed in the image for your application to use.  The missing description and license messages in red will be filled in later.  For now, you have a valid application image located in your Desktops local Docker Repository. 

 

Step 3

 

Lets list out the image you just created in your Desktop Docker repository:

 

docker images

 

 

Step 4

 

Its time to take this image and run it as a virtual machine inside a Docker Container.  With Docker, this is simple to do using the RUN command.  Replace dockerID shown with your own.

 

docker run -p 49160:8080 -d pklein011/node-web-app

 

 

Again, the -d above tells Docker to disconnect from the Quick2Cloud Sample Application.  Should you leave the -d out then the DOS Prompt would become your sample applications message console.  Notice you defined a Container Port 49160 that maps to the applications Port 8080.  You will be using Port 49160 to communicate with your containerized application.

 

Step 5

 

You can always see your console messages by issuing the Docker logs command.

 

docker logs 4d4c

 

 

And if you use the parameter --f at the end of the command then it will sit there as a message console providing messages forever.

 

You now have a Docker Container running with your Quick2Cloud Sample Applications image inside.  Lets take a deeper look and exercise your application from a web browser.

Developing in the Cloud using Toolchains

 

In the basic premise of this Cloud-native concept of Continuous Delivery you will want to interact with tools provided to you online in the Cloud for development, test, and deployment and for placing new features into applications non-disruptively.  In the Cloud this is done using Toolchains, a group of tools capable of achieving this methodology that are freely available for you to use. 

 

The best way to think of Toolchains is conceptually, they are a string of configurable tools, tethered together by a virtual thread making them part of a Toolchain group.  Further, think of Toolchains as themselves running applications, hosted in a special non-chargeable area of the Cloud which you interact with, assisting you to develop, test, deploy and implement your Continuous Delivery strategy.  

 

Tip: Toolchains become your application; they run your repository code under their watchful eye.  Don’t look in the IBM Dashboard for your sample qvsample application name any more like you did when you pushed it up locally from the desktop.   Your application now will become part of the chain.  It will take on the Toolchain name that you now observe in the Dashboard when looking for your application’s information.

As you can see, the IBM toolchain Git Repository is the center of the universe when making Continuous Delivery work.  It is populated with your sample application through a git Clone from the repository located at Github.com and that in turn was populated by you from your local desktop through a git push across the Internet over to www.github.com. 

 

You can actually do a git push from your local desktop directly into the IBM toolchain repository and that technique will be used after the initial clone that the Toolchain does from Github.com.  Github.com will seed the IBM Git Repository with your ver00 code but all development after that, whether at the desktop or in the Cloud, will be committed, then pushed directly to the IBM repository.  A bit later Quick2Cloud will show you how to implement a SSH Key for permission to push from the desktop directly into IBM’s Cloud.

 

Tip: There is another reason to use Github.com while using the Quick2Cloud product.  All the Clouds supported by Quick2Cloud can easily pull your code from Github.com. 

 

The Toolchain Group that has been selected by Quick2Cloud contains the following tools:

 

1.       Issues Tool

2.       Git Repository

3.       Eclipse Editing Tool

4.       Git Editor

5.       Pipeline Delivery Tool

 

What this course will supply is a simple Continuous Delivery methodology that uses these tools in a unique Quick2Cloud way as to give you maximum flexibility, safety and confidence to deploy an application maturely in the Cloud.  Let it be stated that with these tools IBM has given you have the flexibility to be used in various ways and after graduation you may want to explore some additional variations.

 

What this course will give you are two solid ways that are safe and simple to implement and use.  When you take the courses in the advanced college series you will see how to exploit the Toolchain even further., mostly around the Pipeline Delivery tool that does much of the heavy lifting.  In fact, in your Cloud pursuits, the phrase Pipeline Delivery will come up a lot for it plays important roles in containerization concepts that you will cover in the 200-college series courses.  Of all parts of the Toolchain, Pipeline Delivery does the heavy lifting tasks of Build and Deploy for your application.  It takes the git repository as input and delivers an executable that gets placed in the Cloud to run.

 

Continuous Delivery Methodology

 

You will start by creating two Toolchains, one for Production and one for Development.  Each Toolchain will deploy and execute your code in the IBM Cloud, providing you with two URLs, one for each chain.  The code the Toolchain executes depends on the repository the Toolchain is configured with. 

 

In our course, The Production Toolchain will be running off the master branch of the Production Toolchain repository and the Development Toolchain will run off the master branch of the Development Toolchain repository.  The first time your Production Toolchain runs it will get a clone of the repository from Github.com.  The Development Toolchain will also be configured to clone from Github.com in the same fashion.

 

Quick2Cloud places a lot of walls around accidentally altering the Production Repository from misguided or malicious commits and against feature additions that have not been sufficiently tested.   Having your Production code running on its own Toolchain physically allows for this protection while all development and testing takes place elsewhere on the Development Toolchain; providing the safe separation required.  

 

This need for a Toolchain wall will become more apparent when the course discusses the editors and their working relationship to the Toolchain Repository and the trouble you can get into if not careful - especially if you only configure with one Toolchain.  For now, leave the Production Toolchain aside, except for the fact that it is running your Production code every one of your Customers are using, you really won’t touch that Toolchain until its ready to be updated with the most recently tested development features.   

 

Toolchain Editors

 

It is time to consider an editor tool that will work with the Toolchain’s git repositories.   IBM gives you two choices, Eclipse and Git itself.  You should really decide on one and stick with it, if you use both in your development it will only inject problems you will be finding yourself spending hours undoing.  Think of the editors as designed for two different purposes.  Eclipse should be used for developing new features in your application because it has some important capabilities for that purpose.  Git, if used at all, is for small bug fixes in the Production code.

 

Tip: Do not edit with both editors on the same Toolchain!  It will only bring problems.  You will get into a repository out of sync scenario also known in repository terms as a fast-forward error.  Pick one editor and leave the other behind.

 

Eclipse Editor

 

When you bring up the Eclipse Toolchain editor it will make a clone of the Toolchain’s git repository into its Eclipse Workspace.  Editing will be off this Eclipse Workspace Repository.  In addition, every file you modify, add or remove; those modifications will be part of the working directory in the Eclipse Workspace. 

 

(Think of the Eclipse Repository as your local git repository on the desktop.  Think of Eclipse’s modified files as files in your desktop’s working directory). 

 

Once you make changes in the Eclipse Working Directory you will need to commit them to the Eclipse Workspace Repository.  These committed changes will not get reflected in the Toolchain’s Pipeline Delivery Tool (and hence; won’t automatically update the application in the Cloud) until you push the Eclipse Workspace Repository into the Toolchain’s git repository.   See diagram below.  Changes you want to run immediately require a two-step process (commit and push).

 


Spoiler Alert

 

However, you can run the contents of the Eclipse Working Directory (just as you did when running locally at the desktop) by instructing the Eclipse editor to bypass the required commits and pushes, allowing it to run your application straight out of the Eclipse Working Directory for the purpose of testing a feature before committing it.  Later on, this course will demonstrate how to do all that in detail.

 

Note: The first time you bring up Eclipse, its Eclipse Repository will issue a git clone against the Toolchains’ git repository.  Each additional time you start Eclipse in the Cloud, you will be working in the now existing Eclipse Repository - it does not re-clone the Repository from the Toolchain.  The transfer of files from the Eclipse Repository to the Toolchain’s repository, for the Pipeline Delivery Tool to get triggered, happens only through the git push from Eclipse!

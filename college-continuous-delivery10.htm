Quick2Cloud College - Course 104

 

Implementing Continuous Delivery for your Cloud developed application

 

Your courses have gotten you to the point where your sample application is running in the IBM Cloud and you have a Cloudant database Service provisioned, credentialed and bound to that application.  This alone is a major step forward to enjoying the fruits of Cloud computing and you will be learning a lot more on the many flavors of capability you have at your disposal when you study additional courses in the college series. 

 

As you proceed in Course 104, you will see a lot of detail on how Continuous Delivery is accomplished.  Don’t fret, there is a lot here but Quick2Cloud will assist you when the time comes and it will be fairly simple to accomplish.  For now, lean back and relax, we will begin this topic by exploring the remote repository concept, making it the foundation of Continuous Delivery and everything that it provides. 

 

Continuous Delivery is many things which make it a big topic.  First, it can be looked at as a methodology for developing, testing and releasing software in a non-disruptive and automated way to your Customers.  Another view of Continuous Delivery is as a set of productivity tools to accomplish the above methodology; tools that are available to you at no cost as part of your consumption of the IBM Cloud. 

 

You have already studied that repositories, with their own applied methodologies, allows for the development of application features with strict adherence over administrating those features, avoiding the pitfalls that software development historically has shown; pitfalls around regression and poor quality. 

 

But, in your studies, all that repository methodology was introduced to you locally when development was done at the desktop.   The question to be asked, what would a similar methodology look like when development is done in the Cloud to exploit a continuous deployment strategy?

 

Continuous Delivery is not new

 

Using your smartphone you may have already experienced this strategy in some of the apps you have been interacting with.  Take a navigation app, on Monday it gives voice guidance by telling you to turn right on Main Street.  On Tuesday, not only is it telling you to turn right but now giving visual cues such as; a coffee shop is located on the corner.  You say to yourself, when did that feature get put in? 

 

The answer would be Continuous Delivery and its non-disruptive ability of updating the application’s server without the Customer even being aware.  With this kind of constant delivery methodology, features to an application can be added when required, stealthy as Customers use the product with the developer reaping the rewards of an application that keeps getting more functional by the day.  In looking at the history of software development, every six months was a typical duration between new feature updates but now with Continuous Delivery, it can be as often as several times a day! 

 

The reason you want your application to use this methodology is so you too can quickly add features as demanded, making your application’s offering very responsive to your Customer’s needs (plus, you want to look good.  Quickly developed features are becoming the expected norm for today’s Cloud software solutions).  Fortunately, IBM Cloud has a number of tools that allow you to accomplish this methodology and you will be making use of them in this course. 

 

In this topic there needs to be a starting point and ours will be with the creation of a remote repository where your sample application will externally reside.  For this course that repository will be housed at www.github.com  but when it comes time for your own application after graduation, you can choose another repository location.   Think of github.com as an internet connectable website for hosting repositories, a site that is accessible from your desktop and from a vendors’ Cloud which makes the copying and eventual reading back network friendly for both parties. 

 

Tip: Until you are comfortable with your knowledge in your Cloud choice, keep your code external at github.com; at least until you fully understand the Clouds own repository workings.  It’s easy to recover, using Github.com, if you make mistakes while learning and tinkering with the Cloud

 

Step 0

 

Login to www.github.com and create a q2c-web-app repository.   Remember, the gitID you will be using at Github.com is not your ibmID.  If you happen to make both of them the same name, then it becomes easy.

 

 

Click on the New button to create the repository.  The Owner name should be reflecting your github.com gitID

 

 

Notice that your Github.com Owner name is in front of the Repository Name.  When this repository is created it will be known as:

 

https://github.com/gitID/q2c-web-app.git 

 

It should be noted that you can create any number of repositories under your gitID.  Make sure you give all repositories  a description as well plus start with making the q2c-web-app repository public, don’t worry, it’s just your sample application.   Later you can make it private if you like. 

 

Click on Create repository

 

Step 1

 

Great! Now you have a remote repository to pair up with your local one.  Open your BASH console on the desktop and change it to your q2c-web-app directory (you can use the DOS Command Prompt as well).   Issue the following commands:

 

Tip: In the commands below, replace gitID with your actual github.com gitID you created when you signed up

 

Note: The very first time you issue a git push command to GitHub.com you will be prompted for your GitHub userid and password.  After you reply correctly to that prompt you won’t be asked again.

 

These commands will prep the local repository for the copy then the push will do the actual copy.  You will only be copying the master branch of your local repository which is all that is needed for this course.  In reality, what you will be doing is not so much a copy but a replacement of changed files leaving unchanged files alone.

 

Issue these commands in the BASH Console.  Notice there are the above 3 separate commands issued in the display below.

 

git remote add origin https://github.com/gitID/q2c-web-app.git

 

git remote -v

 

git push origin master

 

 

Fantastic!  You have a remote repository at github.com holding a copy of your local q2c-web-app repository.  Pat yourself on the back; you have just passed the first milestone to Continuous Delivery.

Appsody - Build your Sample Application Image with a Kabanero Stack

 

While working in the Appsody development environment, your highly reliable and performing Kabanero Image was operating in a special mode that made development faster and somewhat easy to debug.  This occurred because the Appsody environment has in place special code for this purpose.  However, when it comes time to run this code in a container for production, this Appsody environment must be removed.

 

Appsody uses a Build Command to create a standardized Docker image to be used in a Docker Container and that means your Appsody work can run on any system that supports Docker or in the IBM Cloud in a Kubernetes Cluster.  To take your sample application out of the Appsody development environment and build it standalone you will issue the following command from the q2c-web-appsody/sample2cloud directory.

 

Step 10

 

appsody build .  (don’t forget the period at the end)

 

 

Notice the name in yellow above (dev.local/sample2cloud).  This is the name of the built Kabanero Image for your sample application that resides in the local Docker Repository.  Let’s look at the image in the Docker repository.

 

Step 11

 

docker images

 

 

Even though you developed the code with Appsody, once you built it with the Appsody Build Command it became a normal Docker Image that can be run in a Docker Container.  Let’s run that Kabanero generated image as a Docker image.

 

Step 12

 

docker run -p 29001:8080 dev.local/sample2cloud

 

Note: Use localhost:29001 from a web browser to see your sample application’s home page

 

 

Now that you have a genuine Docker Image from Appsody, and have proved it from the display above, you can use this Kabanero developed image anywhere you run Docker as well as in the IBM Cloud inside a Kubernetes Cluster.

Codewind - an alternative to using Command Prompts

 

Next, let’s look at using your Eclipse Editor, instead of the command line interfaces (DOS Prompts), to manipulate your sample application with Appsody.  It is just one more way of streamlining your development environment by not having to remember and type commands.

 

Codewind is an Eclipse plug-in that can be found in the Eclipse Marketplace.   Locate the Marketplace from the Help menu.

 

 

Using the Find input field, enter in the name Codewind.  Then install the Plug-in.

 

 

Locate the installed Codewind feature via the following Eclipse navigation steps.  Note: This navigation may be different depending on your level of Eclipse.

 

 

Then select Codewind Explorer from the menu.

 

 

Double-click on the Codewind branch of the tree to start it running.  You should see the word Running when it is active.

 

Add your existing sample2cloud project to Codewind.  Click on Add Existing Project and Walk through the Codewind wizard and fill in your sample2cloud project location.

 

 

Codewind will start your Appsody Container using a Port that it picks.  You can find that Port address by looking at the running Docker Containers.

 

docker ps

 

 

In this case your sample application can be contacted on Port or localhost:32802.  You also can see that Codewind added two additional Containers to reside along side yours. These represent added value that Codewind brings to your development environment.  One of the Containers is Codewind’s Performance Dashboard and the other handles the Application Monitor.  There is a Debugging feature using a Port interface for your application as well.  These Containers become active the second you start Codewind in Eclipse and are removed when you stop Codewind in Eclipse.

Things to be careful about

 

Do not use Appsody Commands in conjunction with Codewind or things will get confusing and not work as you would expect.  When Codewind issues its commands under the covers it only knows about those Containers.  If you issue Appsody commands on the side, Codewind will not know about the Containers.  Use either Appsody commands or Codewind for your development but not both at the same time.

 

Keep in mind that starting a 2nd to nth Container will not replace the previous one but starts new ones.  Very quickly it can get confusing to which Container is doing what.  A good practice is to stop Containers after you are done using a docker stop or appsody stop command. 

 

If you start a Container and it fails due to “Port in use” issues you can be sure another Container has that Port.  Stop the older Container then restart your latest Container and it should now run.

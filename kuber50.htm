Clustering Containers using Kubernetes

 

Run Docker Image Directly in Kubernetes

 

Quick2Cloud will show you a faster way to deploy an image into a Kubernetes Cluster when using the IBM Cloud.  This assumes you have done the course on Docker and followed Quick2Cloud’s assists in building a Docker image of your sample application; then pushed that image to the IBM Cloud Registry.  If you did not do the course on Docker, then you can skip this section.

 

 Step 17 (skip if you did not do the course on Docker)

 

Ibmcloud login -u ibmID@email.com

 

List out your IBM Cloud Registry to locate your Docker created Quick2Cloud Sample Application image.

 

 

In the above registry, the entry in yellow is the image name that was pushed to the IBM Cloud Registry in your Docker course.   Again, the first part of the image name is the IBM Cloud Registry, the second part of the name is your namespace (see Docker course) and the third part of the name is the application name.  The names in the registry that start with sample-containers … are the image names generated by IBM Cloud when you used the Kubernetes Toolchain in the previous section to place your sample application in the Custer.  You can ignore them for now.

 

In the previous section, it took many minutes for the Kubernetes Toolchain to process your repository code and convert it to an image.  You can do it far faster using Docker directly.  Therefore, Quick2Cloud will show you how to get your Docker image running in Kubernetes without going through the Toolchain Build and Containerize stages.

 

Step 17A (skip if you did not do the course on Docker)

 

You will configure the Deploy Stage in your Kubernetes Toolchain to use one of the images created by you previously using Docker in your Docker course.  This is to demonstrate that a deploy can happen much faster if the input is an image and not source code.  In this use of Deploy, the Toolchain will not use your Github.com Code Repository or Containerization as input but rather the Docker Image that you created manually and placed in the IBM Cloud Registry using the Docker facility.

 

Let’s configure Deploy.

 

 

In the configuration display below, you will need to update two sections.  Look at the yellow markings and make the changes as indicated.

 

 

You will leave all the other parts of the Toolchain intact as you make these changes to the Deploy Tool only.  Change the Pipeline image version to the latest (in this case 2.5).  Next, highlight and delete any Deploy script that exists.  Then copy and paste a new Deploy script in its place.  That script is listed below.  Replace everything and in the color, purple, replace with the specific names you created in the courses.  What is shown in the above yellow names are ones created for the demonstration.

 

It is time to look at your cheat sheet for you will find all the names you created earlier must be filled in the script below.  As an example:

 

Image name could look like sample2cloud

Image tag could look like latest

Registry URL could look like us.icr.io (or whatever IBM assigned you)

Registry namespace could look like ibmID-myimages (or whatever you picked)

Pipeline Cluster name could look like ibmID-mycluster (or whatever you picked)

 

 

New Deploy script below.  Cut and Paste everything in green and substitute everything in purple.

 

IMAGE_NAME=your-application-name

IMAGE_TAG=your-application-tag

REGISTRY_URL=us.icr.io

REGISTRY_NAMESPACE=your-namespace-name

USE_ISTIO_GATEWAY=

KUBERNETES_SERVICE_ACCOUNT_NAME=

KUBERNETES_MASTER_ADDRESS=

KUBERNETES_MASTER_PORT=

KUBERNETES_SERVICE_ACCOUNT_TOKEN=

IMAGE_NAME=your-application-name

IMAGE_TAG=your-application-tag

REGISTRY_URL=us.icr.io

REGISTRY_NAMESPACE=your-namespace-name

PIPELINE_KUBERNETES_CLUSTER_NAME=your-cluster-name

CLUSTER_NAMESPACE=prod

source <(curl -sSL "https://raw.githubusercontent.com/open-toolchain/commons/master/scripts/check_and_deploy_kubectl.sh")

 

Click on Save at the bottom

 

Let’s run Deploy again.  This will place your Docker Image as specified in the script above, into the Kubernetes Cluster bypassing the Build and Containerize Stages.  Click on the button below in yellow to re-run the Deploy stage.

 

 

You can use Deploy in this way to put any Docker Image you create with Docker commands into a Kubernetes Cluster.   By editing the script above, substituting another Docker image name from the IBM Cloud Registry, that image too can be set to run as an additional Cluster Pod.

 

Reverting Deploy

 

Should you want to reconfigure this Deploy to work by using your Toolchain again, putting back into play Build and Containerize, just delete the above script and add just one line as shown below as your new script:

 

source <(curl -sSL "https://raw.githubusercontent.com/open-toolchain/commons/master/scripts/check_and_deploy_kubectl.sh")

 

And don’t forget to change the Pipeline image version in the setting sheet back to Inherited from Configure Pipeline.

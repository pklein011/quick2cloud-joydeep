Implementing Continuous Delivery for your locally developed sample application

 

Defining the Toolchain Repository as a remote to your local repository

 

Now that you have connectivity from your desktop directly into your Toolchain Repository in the IBM Cloud you need to define that Repository as a remote to your local one.  This will allow you to push your commits directly to the Toolchain without having to type in the entire cumbersome Toolchain Repository name.

 

Your Toolchain Repository has a specific format and always begins with your Toolchain’s region.  In this course it is us-south but it could be different depending on the part of the world you are in.

 

Your repository is named:

 

ibmID is the first part of your ID (not the entire name).  For example, if your id is pklein011@gmail.com then your ibmID for these URLs are pklein011.

 

Notice the first part of the URL is your region, the ibmID is your IBM and the last part of the URL is the specific repository name. 

 

https://us-south.git.cloud.ibm.com/ibmID/simple-toolchain-ibmID-development.git

 

When using SSH, which is how you are going to communicate with this repository, that URL is represented in git protocol as:

 

git@us-south.git.cloud.ibm.com:ibmID/simple-toolchain-ibmID-development.git

 

Thus, the git protocol URL for pklein011’s ibmID would look something like

 

git@us-south.git.cloud.ibm.com:pklein011/simple-toolchain-pklein011-development.git

 

Step 5

 

You will define this remote repository name to your local repository.  This is accomplished with the use of a git command especially for this purpose.  In the q2c-web-app directory, issue the following command (replacing the pklein011 ID with yours).  Don’t make any typos!

 

git remote add qvprod git@us-south.git.cloud.ibm.com:pklein011/simple-toolchain-pklein011-development.git

 

git remove -v

 

 

Notice, that you now have two remote repositories defined.  One is defined for Github.com (origin) and the other for the IBM Toolchain (qvdev).  Also, notice that your Github.com repository is defined as a URL and not with git protocol.  The push that will go to Github.com is not using SSH but rather HTTPs protocol.  For IBM Cloud, your most secure way of updating your Toolchain repository is by using SSH.

 

Step 6

 

It is time to make your first push from your desktop repository to the Toolchain over SSH.  First, make sure your latest changes have been committed to your local repository.  Also, make sure you have tested those changes out by locally running it and applying some form of unit tests.  When you are sure you have a good set of features to deliver to the Cloud you will do the following.  Make sure you are in the q2c-web-app directory.  Issue the following:

 

git push qvprod master

git push origin master

 

You will be prompted for your passphrase, then your most recent commits will be transmitted over SSH to the IBM Toolchain and that Toolchain Repository will be updated with your most current features.

 

Note: If you have not changed anything since you created your Github repository then what you have on the desktop and in the IBM repository are the same.  In this case you will get a message saying everything is up to date.  This means the push had nothing to do.

 

As part of this process, the Toolchain’s Pipeline Delivery tool will detect the update and proceed to automatically redeploy your application by first renaming your old application, starting the new one, then deleting the old one after the new one starts.  In operating in this fashion, your Customer’s will not observe any kind of interruption.  Since in a prior part of this course you added your Services and URL to the deploy script – everything defined to your application remains intact.

 

You will also be pushing an additional copy to the Github.com remote repository so you have a backup and one to use later for the other Clouds should you choose to deploy elsewhere.  It’s good to keep both repositories at the same commit level so its best practice to push to both when you push your features to the IBM Toolchain.

 

Note: In utilizing all the work you have accomplished in Course 104, you are now in position to update Production with your application’s latest version

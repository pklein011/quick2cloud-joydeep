Binding Services using VCAP Services

 

With your Cloudant and Watson Services now bound with your application your program will need a lot of credential information before it can start using them.  The idea of having you hardcode this information into your application, using Cut and Paste from a file, is a task that the Cloud wants you to avoid.  For one reason it increases the risk your credentials will be exposed and its problematic because credentials may change making you have to keep track of where the new credentials are to be re-pasted.

 

To accomplish getting this credential information programmatically into your application, the Cloud passes the credentials as an Environment Variable filled with JSON structures by the name of VCAP_APPLICATION and VCAP_SERVICES.   Your application is highly requested to use a function that parses those variables to return with things your application needs like the Cloudant URL, userID and password as well as Watson’s URL and API key

 

Unfortunately, there are many ways to parse these VCAP variables and developers have used many a technique and willing to share with you those techniques with just a search on Google.  But there is an issue with all these techniques; they require two ways of parsing the variables:

 

1.       A technique for parsing in the Cloud

2.       A technique for parsing on the desktop

 

In this course you will address a technique that provides one clean way to parse the VCAP variables, in the Cloud as well as locally on the desktop; using just one piece of code.  In this way you can test your application locally then push it to the Cloud without making any additional code changes.  Quick2Cloud’s technique uses the CFENV facility; available as a function to Node.js and other languages. 

 

By default, CFENV looks for the VCAP_APPLICATION and VCAP_SERVICES variable that is passed in by the Cloud and provides a nice set of functions to parse them.  But, as you will see, CFENV will not find a VCAP_SERVICES environment variable when running the sample application locally on the desktop.  That VCAP variable is only maintained and passed when running in the Cloud.

 

Quick2Cloud solves this problem by giving you a q2cloud.js command that takes the Cloud application’s VCAP variables and copies them to a local desktop file called qvsample-vcap.json. When the q2cloud command is run, two additional files are created in the q2c-web-app directory called VCAP_APPLICATION.json and VCAP_SERVICES.json which are mirror images of the respective VCAP variables.

 

It is accomplished by issuing:

 

1.       A cf env IBM Cloud command

2.       A q2cloud Quick2Cloud command

Tip: The sample CFENV application code that makes use of these two files can be lifted out and used in your other applications as well.  For now, this course will show how it is used for the sample application.

 

At this point, the CFENV facility has all the information it needs, just as it would if running in the Cloud.  All coding going forward is the same for the Cloud as it is for the desktop.  It should be obvious that when you are running in the Cloud the two locally generated VCAP files would not be read and CFENV would be using the variables passed to it by the Cloud.   You would need to run the above script each time you add a Service to your application so the locally developed version of that application can get the latest VCAP information.

TIP: it is important that no one outside of your development community sees or gets access to VCAP_APPLICATION.json or VCAP_SERVICES.json files or they will be able to get access to your data from their application. 

 

Therefore, these files need to be locked down from causal intruders but don’t encrypt, the sample application won’t be able to retrieve them if you do.
